## Resolved Issues

The following table lists the resolved issues for HPE CSI Driver for Kubernetes v3.1.0.

| ID |Component |Title|Resolution |
|----|----------|-----|-----------|
|CON-2683|csp.3par|Duplicate IQNs shouldn't be allowed in hpenodeinfo|Detect duplicate IQN/WWN/NQN and fail csi node daemon|
|CON-3070|csp.3par|Users needs to be able to control log level on 3PAR CSP|Added LOG_LEVEL environment variable for primera CSP deployment|
|CON-3165|csp.3par|CSI v2.5.2 - Unable to expand a PVC when added to RCG|For Primera and Alletra 9K, stop RCG, expand and restart RCG.|
|CON-3870|csi.k8s|Make /dev relocatable for backup tools|Using mknod to create the raw block devices.|
|CON-4174|csi.k8s|Unable to restore applications with Veeam Kasten|Allowing import of volumes with pvc- prefix|
|CON-4194|csp.3par|Snapshots are created only on the primary storage array side|If volume replicated, set 'SyncSnapRCopy' to true to create snapshot on replication target array as well.|
|CON-4206|csp.3par|Volume gets cloned as wrong provisioning type|PVC Created from shapshot always takes parent volume provisioning type|
|CON-4208|csp.3par|datasourceAsPVC for a tdvv volume fails|For Offline clone, only TDVV was being considered, added reduce and Dedup|
|CON-4212|csp.3par|Volumeinfo CRD not getting updated if a PVC is removed from APP|Updated the Volumeinfo CRD upon pvc removal|
|CON-4235|csp.3par|ImportVolumeName and importVolAsClone fails due to "pvc-" prefix.|Removed the check, since replicated volumes can be imported into foreign clusters.|
|CON-4250|csp.3par|Publish of restored snapshot cloned volume fails in Active-Active PP|Update the CRD with RCG Policy when volume is cloned from replicated APP policy.|
|CON-4285|csp.3par|Issues restoring Velero backups"|Removed the check, since replicated volumes can be imported into foreign clusters.|
|CON-4294|csp.3par|Snapshotting fails with async replication (virtualCopyOf/datasourceAsPVC/volume snapshot)|Check mode of replication before setting syncSnapRcopy parameter|
|CON-4385|csp.3par|provisioningType as reduce fails with Alletra9k version "9.6.20.6"|Updated the Alletra 9K software version number|
|CON-4423|csp.3par|Statically provisioned PVC in v3.0.2 - Cannot restore from snapshot/delete snapshot|Updated the VolumeSnapshotContent CRD with proper restore size|
|CON-4431|csp.3par|After CSI Driver upgrade from 2.5.1 to 3.0.2 Old PVC's are not getting mounted to POD|Ignoring case and accepting StorageClass parameter accessProtocol|
|CON-4439|csp.3par|Setting replicationPolicy in mixed/upper case falls back to Classic PP instead of throwing error|Ignoring case and accepting Active policy|
|CON-4477|csp.3par|Delays in PVC clone to get into bind state after upgrade to 3.0.2. CSI Driver waiting for actual vvcopy data movement to complete before binding cloned PVC|For online copy, background operation is set to Completed, and the PVC moves to the Bound state immediately.|

## Known Issues

Please note that, `Known Issues` from previous releases are still applicable with suggested workarounds, if they are not part of `Resolved Issues` above.
| ID | Component | Title | Description | Workaround |
|----|-----------|-------|-------------|------------|
|CON-4441|csi.k8s|Inline ephemeral scrubber traversing entire filesystems|Scubber traverses directories beyond the mount point and spews errors|None|
|CON-4399|csp.3par|200 PVC/Pod creation shows multiple CSP errors, panics and PVC takes 10+ min to all bind|Multiple clients cause issues both in CSP and array end, eventually all PVCs get created and bound|Eventually all PVCs and Pods will get created|
|CON-4465|csp.3par|NVMe with multiple vlan tagging for all ports appends * to ips during discovery and fails|The aterisk causes NVMe discovery failure|Configure at least 1 port  with single interface (either untagged or tagged)|

|CON-4448|csp.3par|invalid replicationPolicy still falls back to classic PP in AlletraMP|Invalid replicationPolicy values in StorageClass is not erroring out|Always set replicationPolicy to "active" if active peer persistence is required|
|CON-4193|csp.3par|VM stop and subsequent start does not work in case of primary storage array failure|In case of primary storage array failure, if the VM is stopped, subsequent VM start fails|VM restart works fine when primary array is unreachable|

|CON-4254|csp.3par|Replicated snapshot is not deleted from the secondary array|When an APP volume is snapshotted and deleted, the snapshot is only deleted from Primary array and not the secondary array.|The snapshot can be manually deleted from the secondary array.|
